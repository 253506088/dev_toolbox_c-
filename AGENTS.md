# 通用礼节 (General Etiquette)
- 优先保证代码简洁易懂。
- 别搞过度设计，简单实用就好。
- 写代码时，要注意圈复杂度，函数尽量小，尽量可以复用，尽量不写重复代码。
- 写代码时，注意模块设计，尽量使用设计模式。
- 写代码时，注意拆分文件，按照需求进行拆分，尽量一个文件内不要同时多个需求的实现，除非是共用组件。
- 写代码时，将日志写好写详细方便出问题调试，日志用中文写。
- 写代码时，注意写好注释，用中文写，每一个类、方法、函数都需要用注释说明该类/方法/函数是干什么的，以及复杂逻辑的代码段都需要做备注说明
- 进行代码修改时，严禁调整任何无关的格式、空格、换行或注释。只针对目标逻辑进行最小化变更，保持原有代码风格完全一致
- 给我解释代码的时候，说人话，别拽专业术语。最好有图（mermaid风格）
- 帮我实现的时候，需要给出原理，并给出执行步骤，最好有图（mermaid风格）
- 改动或者解释前，最好看看所有代码，不能偷懒。
- 改动前，要做最小化修改，尽量不修改到其他模块的代码
- 改动后，假定10条case 输入，并给出预期结果
- 给出的mermaid图，必须自检语法，可以被渲染，在暗黑主题上清晰可见
- 给出的mermaid图，必须要可以被暗黑主题渲染清晰
- 给出的mermaid图，必须要注意中文的描述增加双引号""包裹起来
- 给出的mermaid图、给出的文档，统一放在当前项目根目录下的my_doc目录下，如果不存在my_doc那就创建一个然后再写

# 需求处理流程 (Requirement Protocol)
- 收到新需求时，请先自行评估复杂度。
- 判定标准：涉及超过3个文件修改、引入新库、或涉及核心架构变更的，视为【复杂需求】。
- 处理方式：
    - 【复杂需求】：
        - 必须先分析当前项目的状况，输出技术方案或Todo计划的md文件，要用mermaid图来画出逻辑流程，询问我“是否按此执行”，待我给出明确的开始编码许可后方可写代码。
        - 如果有拿不准、可能存在问题的点，你要专门写出【等待用户确认事项】，在里面写明白具体的情况，由用户来抉择。
        - 复杂需求很难一次就把计划写完美，我们可能会来回沟通打磨数次才会开始编码，所以不着急马上干活，等用户的明确的开始编码命令。
        - 【简单需求】：直接按最佳实践实现，无需确认。

# 实验性规则 (Experimental Rule)
当你被要求修复一个 Bug 时，请遵循以下步骤：
1.  **理解问题 (Understand):** 仔细阅读 Bug 描述和相关代码，复述你对问题的理解。
2.  **分析原因 (Analyze):** 提出至少两种可能的根本原因。
3.  **制定计划 (Plan):** 描述你打算如何验证这些原因，并给出修复方案。
4.  **请求确认 (Confirm):** 在动手修改前，向我确认你的计划。
5.  **执行修复 (Execute):** 实施修复。
6.  **审查 (Review):** 查看自己的修改有没有问题。
7.  **解释说明 (Explain):** 解释你做了哪些修改以及为什么。


## 开发规则

- 优先保持代码结构清晰，不要把所有逻辑塞进一个类。
- 涉及游戏运行时对象修改时，必须先做 null 检查和当前状态检查。
- 不要随意重命名已有 public API，除非用户明确要求。
- 不要删除用户已有代码，除非确认它确实无用。
- 修改前先解释改动思路，再动代码。
- 每次完成后说明改了哪些文件，以及为什么改。
